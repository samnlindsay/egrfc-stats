# Agents guide

This document acts as your onboarding pack when working in the `egrfc-stats` repository. It explains the project purpose, key components, workflows, and the house rules you must follow in Python, HTML, JS and Markdown.

## Project overview

The `egrfc-stats` project is a comprehensive data analytics and visualization platform for the East Grinstead Rugby Football Club (EGRFC). It ingests raw data from various sources, processes and enriches it in a DuckDB database, and serves it through a frontend website with interactive charts and tables. The project is structured into several key components:

- **Data Ingestion and Processing**: Python scripts that scrape data from Google Sheets, Pitchero (egrfc.com) and englandrugby.com, clean and transform it, and load it into a DuckDB database. This includes handling historical data, managing caches, and ensuring data quality.
- **Backend Database**: A DuckDB database that serves as the canonical source of truth for all data used in the frontend. It is built and updated through a defined pipeline, with attention to lock-safe workflows for concurrent access. Tables are designed for efficient querying and support the various frontend features, aggregating the available data at various levels of granularity (player, game, appearance, season, etc).
- **Frontend Website**: A collection of HTML, CSS, and JavaScript files that render the user interface of the site. It includes pages for player profiles, team sheets, and a database explorer. The frontend fetches data through pre-exported JSON files and visualizes it using Vega Lite charts and HTML Bootstrap components. It is designed to be responsive and user-friendly, with interactive filters and tooltips to enhance the user experience. The majority of users will access the site through their mobile devices, so mobile-first design principles are followed. The frontend is built to be static and served through GitHub Pages, so all dynamic data interactions are handled through pre-generated JSON files that are updated as part of the backend build process. Users can also view (and potentially query) the underlying DuckDB database directly through a Data page on the site, which provides a read-only interface to explore the tables and download data for their own analysis.
- **Charts and Visualizations**: A set of predefined Vega Lite chart specifications that are rendered on the frontend to visualize key metrics and trends. These charts are designed to be informative and visually appealing, providing insights into player performance, team dynamics, and historical trends. The chart specifications are stored as JSON files in the `data/charts` directory and are loaded dynamically by the frontend JavaScript code.
- **Documentation and House Rules**: Markdown files that provide documentation on the backend schema, data sources, and coding standards. This includes guidelines for writing Python code (e.g., use of type hints, docstrings, and modular functions), HTML structure (e.g., use of Bootstrap classes, semantic tags), JavaScript practices (e.g., event handling, DOM manipulation), and general project conventions (e.g., commit message format, branching strategy). The documentation is intended to ensure consistency across the codebase and to help new contributors understand the project quickly.
- **Operational Workflows**: Defined processes for updating the backend database, refreshing caches, and deploying frontend changes. This includes regular update routines (e.g., weekly updates to keep data current), ad-hoc workflows for handling specific tasks (e.g., refreshing Pitchero data), and guidelines for managing concurrent access to the DuckDB database to avoid lock conflicts.
- **Testing and Quality Assurance**: While not explicitly detailed in the initial documentation, there is an implicit expectation that code changes will be tested for correctness and performance. This includes validating data transformations, ensuring that charts render correctly with the updated data, and verifying that the frontend remains responsive and functional across different devices and browsers.

## Repository structure

- `python/`: Contains all Python scripts for data ingestion, processing, and backend management. This includes the main build script (`build_backend.py`), update routines (`update.py`), and any utility functions or modules related to data handling.
- `data/`: Stores the DuckDB database files, exported JSON data for the frontend, and chart specifications. This directory is organized into subdirectories for clarity (e.g., `data/backend/` for backend exports, `data/charts/` for chart specs).
- `js/`: Contains JavaScript files that handle frontend interactivity, data fetching, and chart rendering. Each page has its own JS file (e.g., `player-info.js`, `team-sheets.js`), and there may be shared utility scripts (e.g., `nav-layout.js`).
- `css/`: Contains CSS files for styling the frontend. This includes custom styles as well as any overrides for Bootstrap components.
- `*.html`: The HTML files that define the structure of the frontend pages. Each page corresponds to a specific feature (e.g., `player-info.html`, `database.html`).
- `README.md`: The main documentation file that provides an overview of the project, instructions for setup and usage, and general information about the repository.
- `AGENTS.md`: This document (the one you're reading) that provides a detailed guide for agents working in the repository, including an overview of the project, its structure, and the workflows to follow.

## Data flow and update process

The data flow in the `egrfc-stats` project follows a defined process to ensure that the backend database is regularly updated with the latest data, and that the frontend reflects these updates through pre-generated JSON files. The main steps in this process are as follows:
1. **Data Ingestion**: Python scripts scrape data from Google Sheets, Pitchero, and englandrugby.com. This includes both current operational data (e.g., recent games, player appearances) and historical data (e.g., past seasons). The scripts handle data cleaning and transformation to ensure that it is in a consistent format for loading into the database.
2. **Database Update**: The ingested data is loaded into a DuckDB database through a build process defined in `build_backend.py`. This process is designed to be cache-first, meaning it will use existing cached data where possible to avoid unnecessary web scraping. The build process can be configured to refresh specific data sources (e.g., Pitchero) if needed.
3. **Export for Frontend**: Once the database is updated, specific tables and views are exported as JSON files that the frontend can consume. These exports are stored in the `data/backend/` directory and are structured to support the various frontend features and charts.
4. **Frontend Update**: The frontend JavaScript code fetches the updated JSON files and re-renders the charts and tables accordingly. This ensures that users always see the most current data when they visit the site.
5. **Operational Workflow**: A regular update routine is defined in `update.py`, which can be run with a single command to perform the entire update process, including building the backend, refreshing caches, and regenerating chart specs. This routine is intended to be run on a regular basis (e.g., weekly) to keep the site up-to-date with the latest data.


## House rules and best practices

When working in the `egrfc-stats` repository, it is important to follow certain house rules and best practices to maintain code quality, ensure consistency, and facilitate collaboration. Here are some key guidelines to keep in mind:
- **Python Code**: Use type hints and docstrings for all functions to improve readability and maintainability. Follow PEP 8 style guidelines for formatting, and structure your code into modular functions that perform specific tasks. Avoid hardcoding values; instead, use configuration files or parameters where appropriate.
- **HTML Structure**: Use semantic HTML tags and Bootstrap classes to ensure a responsive and accessible design. Keep the structure of the HTML files organized, with clear sections for headers, filters, charts, and footers. Use data attributes to store metadata for interactive elements (e.g., tooltips).
- **JavaScript Practices**: Use event delegation for handling user interactions, and keep your JavaScript code organized into functions that handle specific tasks (e.g., loading data, rendering charts). Avoid global variables where possible, and use modern JavaScript features (e.g., async/await, template literals) to improve readability and performance.
- **Documentation**: Update the documentation as needed when making changes to the codebase, especially if you are adding new features or changing existing workflows. This helps ensure that other contributors can understand and work with your changes effectively.

## Frontend conventions and repeated UI patterns

The frontend already has a fairly consistent visual and interaction language. New pages and components should follow these patterns unless there is a strong reason to deviate.

- **Core design tokens**: Shared colours live in `css/variables.css`. The main site palette is dark blue `--primary-color: #202946`, light blue `--accent-color: #7d96e8`, light neutral `--light-color: #e5e4e7`, chart surface `--chart-surface-color: #f2f1f4`, green `--green-color: #146f14`, and red `--red-color: #991515`. Prefer reusing these variables over introducing one-off colours.
- **Squad colour mapping**: The site consistently treats the 1st XV as the dark blue primary colour with light text, and the 2nd XV as the light blue accent colour with dark blue text. Neutral or combined views often use black/white as a third state. Existing shared classes in `css/components.css` already map this through `--team-bg` and `--team-fg`; extend those classes rather than inventing new per-page colour rules.
- **Typography**: PT Sans Narrow is the dominant UI font for page chrome, filters, bootstrap-select controls, form labels, and much of the general interface. Keep that typographic direction when adding controls or page-level UI.
- **Page header pattern**: Content pages typically start with a `.page-header` block containing a title, a short descriptive paragraph, and a right-aligned or adjacent `.page-header-filters` area. Reuse that structure instead of scattering filters deeper into the page.
- **Filter construction**: Filters are usually built from `.filter-item` wrappers containing an `.input-group` with a left-hand `.input-group-text` label and a Bootstrap or bootstrap-select control on the right. Label pills use the primary colour background and white text. Preserve that pattern for consistency.
- **Filter control conventions**: Selects generally use `selectpicker` from bootstrap-select, with `data-size` set for dropdown height and `multiple` used for season or position filters where multi-select is expected. Common option wording is already established: squad filters use `All squads`, `1st XV`, `2nd XV`; game-type filters use `All games`, `League + Cup`, `League only`; multi-select filters use titles like `All seasons` or `All positions`.
- **Dense-data disclosure pattern**: For pages with many charts or tables, the preferred pattern is not to dump everything open on the page. Use the shared `.chart-panel` structure with a `.chart-panel-toggle` button and a hidden `.chart-panel-content` section. Panels start collapsed by default.
- **Accordion behavior**: When panels belong to the same logical section, use `data-accordion-group` on `.chart-panel-toggle`. The shared logic in `js/shared.js` will then keep one panel open at a time within that group. This is the established pattern on pages such as Squad Stats and Performance Stats.
- **Panel colour variants**: The toggle button variants already carry meaning: `.chart-panel-toggle--primary` is used for 1st XV content, `.chart-panel-toggle--accent` for 2nd XV content, and `.chart-panel-toggle--black` for neutral, combined, or supporting views such as tables and aggregate summaries.
- **Squad comparison layout**: When presenting equivalent 1st and 2nd XV content, the default pattern is side-by-side or paired presentation with mirrored headings and cards, followed by an optional neutral `Total` or combined summary. Season Summary and Squad Stats both follow this approach.
- **Metric cards and surfaces**: Summary numbers are commonly shown in cards with rounded corners, subtle box shadows, and a strong coloured header strip. Reuse the existing `squad-metric-card*` classes and related shared card styling rather than creating bespoke card systems.
- **Muted supporting copy**: Section intros and explanatory text are usually a smaller muted grey paragraph immediately under the section heading. Keep those explanations brief and informative.
- **Responsive navigation**: Top-level pages use the shared Bootstrap navbar with a mobile collapse toggle. If a new page belongs in the main site navigation, keep it aligned with the existing navbar structure and `app-top-navbar` styling.
- **Player profile squad treatment**: Player profile cards also follow the squad split, with 1st XV and 2nd XV using different border tints and different headshot background assets. If extending that page, preserve those squad-specific visual cues.
- **Squad Stats filter model (reference implementation)**: Squad Stats now uses an offcanvas-only filter drawer at all breakpoints (season, game type, minimum appearances, unit), opened from contextual filter chips near each section. Do not reintroduce a large top-of-page filter panel for this page.
- **Scoped filter chips**: Active filter chips should reflect only controls that affect the local content block. For example, in Squad Stats, the 1st XV league comparison chips intentionally omit game type and minimum appearances because those controls do not affect that section.
- **Game type default**: Where game type filters include friendlies, default to `League + Cup` so friendlies are opt-in.
- **Section jump navigation**: Long analytical pages should support in-page section jumps with stable section ids and URL hash updates to make links shareable.
- **Tooltip behavior on touch/mobile**: Vega tooltips are disabled on coarse-pointer/mobile contexts in shared embed code to prevent accidental triggers and clipped overlays; desktop keeps hover tooltips.
